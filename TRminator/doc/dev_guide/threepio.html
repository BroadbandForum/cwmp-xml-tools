About The The Threepio Engine

The Threepio Engine is designed to be a small, quick, extensible XML processor focused on the creation of Tables with a row for each item of a (currently) hard-coded "major item type."
There are also hard-coded column names, which are filled by the engine. Unlike the "major item type," it is intended that the user can define and/or re-order these.
To see completed impelmentation of User Interfaces for the Threepio engine, see information on the TRminator, a set of user interfaces created for the Broadband Forum for creating Tables of their Data Models.

The Threepio class

Package: threepio.engine
Description: The Threepio class is intended to be a liason between User interfaces and the lower level classes use for creating Tables.
It contains three methods that carry out useful sets of method calls, in three functions:

* docToTable is a method that creates a Table for a Data Model from an XML document, involving any previous versions of the model, as imported in the newest file. The table is then printed out to a file, in HTML format.
Options include diffing the most recent version and one previous.

* diffTWoTables carries out the function of docToTable for two different versions of the same Data Model, and prints out a table of a diff between the two.

* xmlToPrintedTable prints out an HTML table of the contents of a user-defined "container" from an XML document.

Requirements: none
Uses:
Threepio has methods that work well as "cookie-cutter" ones to call from a UI.
the TRminator relies heavily on Threepio.

Documenter (Interface)

Package: threepio.documenter
Description: The Documenter Interface defines methods used to create Doc objects from raw XML.
These Doc objects are simply queues of XTag objects (a class that models a Tag from HTML or XML) and Strings.
Docs are much easier to work with than the raw XML.

Requirements: Documenters must be able to convert files based solely on their XML, or based on an Entry/ Doublet that has a key of the file's path as a String, and a value of the model version.
Uses:
Threepio uses the XDocumenter class to create XDoc objects (an implementation of Doc), when making Tables out of XML Data Models.

Tabler (Abstract Class)

Package: threepio.tabler
Description: The Tabler converts Doc objects to Table objects.
These Table objects are IndexedHashMaps, with keys of row names, and values of Rows (lists of cells).

Requirements: Tablers must produce some form of Table, with input of some form of Doc. Uses:
Threepio uses a TablerController object, which uses Tabler objects to create the tables that are printed out.

TagHandler (Abstract Class)

Package: threepio.taghandler
Description: TagHandler objects are used to process XML based on a single tag type.

Requirements: TagHandlers must have a type that they handle, and MUST ONLY handle the information that starts with that tag type and ends with a closing tag of that type.
They MUST remove the information they handled. They MUST be named in a way that denotes the type they handle.
The HandlerFactory MUST be updated with this classname. Uses:
Tablers rely on TagHandlers to parse Docs, through use of the HandlerFactory. Extension of GeneralTagHandler is often sufficient for creating a new TagHandler.

Working with Components

Components (used in Broadband Forum Data Models) require extra parsing work. The ModelTabler class, a Tabler, has two getComponents methods that return a Table based on the component objects in the document.

Working with Profiles

Profiles (used in Broadband Forum Data Models) are defined in-line with the rest of the data in the Data Models, but are usually not expected to be in the same table as the rest of the data. For this reason, the ModelTabler class, a Tabler, creates a single row for each profile in a separate Table, which can optionally be included in a table with the Table for the main data.
